Это справочный материал к Этапу 9 — **не команда начинать Этап 9**. Решение о начале, как обычно, за тобой ("Следующий этап").

## Итог (если читать только один раздел)

- Архитектуру менять не нужно — Collector/Storage/Analyzer уже разделены правильно. Единственный пустой кусок — Analyzer.
- Вся надёжность проекта держится на одной формуле (раздел 2) — `sell_price × 0.85 − buy_price`, порог безубытка ~17.6%. Это единственная не-гипотеза во всём документе.
- Исходный вопрос «как отсеивать мусорные лоты» закрылся сам: мусорный лот — это лот с низким `liquidity_factor` и высоким `volatility_penalty` в формуле score, отдельного шага-фильтра не нужно.
- В `docs/PLAN.md` (Приложение А) уже есть черновой дизайн Analyzer — независимо от этого документа, и он почти дословно совпадает. Это не повод бездумно доверять готовности: веса `w1–w4` не подобраны, пороги не заданы, бэктеста не было ни разу — подтверждено Claude Code на 15.07.2026.
- Данных сейчас физически мало (3 захардкоженных предмета, история копится недавно) — этого недостаточно, чтобы содержательно бэктестить `investment`-сигнал уже сегодня.
- `flip` и `investment` не равны по готовности к реализации: `flip` можно тестировать почти сразу, `investment` — только после накопления истории за недели. Это естественная граница, если Этап 9 дробить.
- Steam fee уточнён: **0.15** (5% + 10%), не диапазон 0.13–0.15 из Приложения А.
- Порядок реализации (раздел 12) — сначала расчёт score без доверия к нему, потом бэктест, потом калибровка весов, и только потом расширение охвата и подключение Telegram.

## 1. Термины

«Лонг/шорт» в обсуждении = `SignalType.INVESTMENT` / `SignalType.FLIP` в коде (`storage/models.py`). Не направление ставки (шорта в смысле продажи без владения на Market не существует — нет заимствования, нет маржи), а горизонт:

- **flip** — лот сейчас заметно дешевле справедливой цены, купить и перепродать за часы/дни.
- **investment** — устойчивый тренд роста за недели/месяцы, купить и держать.

## 2. Формула комиссии — единственный не-гипотетический факт в документе

Steam берёт с продажи 15% (5% Steam + 10% издатель; на дешёвых предметах округление до цента ухудшает эффективную комиссию ещё сильнее). Комиссию платит только продавец.

```
net_profit = sell_price × 0.85 − buy_price
```

Точка безубыточности: `sell_price ≈ buy_price × 1.176` — нужно минимум ~17.6% разницы между входом и выходом просто чтобы выйти в ноль, не в плюс. Применимо к `flip` и `investment` одинаково. Приложение А помечало `steam_fee_rate ≈ 0.13–0.15` как «уточнить отдельно» — уточнено, 0.15. Claude Code подтвердил: это именно уточнение запрошенного, а не расхождение с Приложением А.

## 3. Market hold vs trade hold — нужно проверить руками, не только по источникам

Судя по независимым источникам, предметы CS2, купленные на Market, маркетабельны (можно перевыставить) сразу — 7-дневное ограничение касается только P2P-трейдов, не перепродажи через сам Market. Это важно для `flip`: если бы было наоборот, вся стратегия превращалась бы в вынужденный недельный холд с ценовым риском. Источники комьюнити-уровня, не официальная документация Valve напрямую — этого вопроса не было в «Приложении А» вообще, это чистое дополнение. **Перед тем как строить на этом логику — купить самый дешёвый предмет (пара центов) и попробовать сразу выставить обратно.**

## 4. Реальный источник истории цены (сверено с кодом 15.07.2026, повторно подтверждено Claude Code)

- Автосбор (`scheduler/collection_job.py`, APScheduler, каждые 60 минут) использует только `priceoverview` — публичный эндпоинт, даёт текущие price+volume без истории. Список предметов — жёсткие 3 штуки (`TRACKED_ITEMS`): AK-47 Redline (FT), AWP Asiimov (FT), Glock-18 Fade (FN).
- `pricehistory` — не часть автосбора. Отдельный ручной скрипт (`scripts/backfill_price_history.py`), требует личный cookie разработчика (`STEAM_SESSION_COOKIE`), гоняется вручную по тем же 3 предметам.
- `fetch_market_items()` (динамический список через `search/render`) уже реализована в `collector/steam_market_client.py`, но не подключена к `collection_job.py`. Строительный блок для «воронки» есть, сборки в пайплайн — нет.

Следствие: история для `investment`-сигнала сейчас = то, что накопил почасовой сбор с момента запуска Этапа 8, плюс разовый backfill — и только по 3 предметам, не по рынку в целом.

## 5. Анти-бот механика — уже решена, контекст для любых новых функций сбора

`priceoverview`/`search/render` режут «голые» запросы (без cookies и `Referer`) как подозрительные. Решение уже в коде: перед первым запросом — обычный GET на страницу листинга/поиска, как сделал бы любой анонимный посетитель, дальше `Referer` на неё же. Не обход антибота — обычный путь пользователя.

**Уточнение от Claude Code (15.07.2026):** механизм разогрева на деле не единый, а два разных. `_ensure_session_warmed_up()` (используется `fetch_price_overview`) — «умный»: проверяет протухание cookies по `Expires`, греется только если нужно. Внутри `fetch_market_items()` — отдельный инлайн-разогрев, безусловный, срабатывает при каждом вызове, даже если сессия уже тёплая. Не баг, но лишние запросы и несогласованность — стоит унифицировать, когда `fetch_market_items` будут подключать к автосбору (см. раздел 12), а не оставлять как есть при расширении охвата.

Реальный rate limit из `docs/SCOPE.md`: при тесте с интервалом 6 секунд поймали 429, блок держался больше 20 минут — жёстче, чем типичные комьюнити-оценки.

## 6. Пайплайн — что готово, что нет

1. **Collector** (готово) — `fetch_price_overview()` (один предмет), `fetch_market_items()` (список, не подключена).
2. **Scheduler** (готово) — `collect_once()` раз в час по 3 предметам, ошибка по одному не роняет цикл.
3. **Storage** (готово) — `items` / `price_snapshots` / `signals`, миграция Alembic, дедуп снепшотов по времени (±1 минута).
4. **Analyzer** (Этап 9, не начат) — здесь и должно появиться решение «достоин ли лот внимания», по данным из `price_snapshots`.
5. **Signal Generator / Telegram Bot** — не начаты.

## 7. Дизайн flip/investment поверх уже существующей схемы `signals`

Таблица `signals` уже содержит `spread_score`, `trend_score`, `liquidity_factor`, `volatility_penalty`, `score` — Claude Code подтвердил напрямую по `models.py`, все поля nullable. Приложение А уже даёт черновую формулу:

```
final_score = w1*spread_score + w2*liquidity_factor − w3*volatility_penalty + w4*trend_score
```

**Для flip:**
- `spread_score = (median_sell_price_7d − current_buy_price) / current_buy_price` (уже в Приложении А).
- `net_profit` — формула раздела 2, с `steam_fee_rate = 0.15`.
- Фильтр по ликвидности (мин. продаж/день) и по волатильности (`std_dev/mean` выше порога — отсекать) — уже намечены, конкретные пороги не заданы.
- Дополнительно к Приложению А: подтверждение отклонения на двух подряд опросах Collector'а (защита от разового глюка/устаревшей цены) — новая идея, не встроена ни во что существующее, см. рекомендацию в разделе 12.
- Честная оговорка: без автопокупки часть flip-сигналов физически не успеть реализовать (та же аномалия видна другим ботам). Риск — «не успеть», не «потерять деньги»: лот либо купите по выгодной цене, либо не купите вообще.

**Для investment:**
- MA7 vs MA30 и наклон линейной регрессии (`trend_score`) — уже намечены в Приложении А.
- Тот же порог доходности (раздел 2) применяется к ожидаемому росту.
- Требует истории за недели — см. раздел 4, сейчас доступно только для 3 отслеживаемых предметов.

## 8. Бэктестинг — обязателен, независимо подтверждён дважды

Приложение А фиксирует необходимость бэктеста; Claude Code подтвердил, что в репозитории нет ни одного файла/скрипта с бэктестом. Ни `spread_score`/`trend_score`/веса `w1–w4`, ни абсолютные пороги нельзя считать надёжными без прогона на истории. Сейчас полноценный бэктест `investment` невозможен физически (мало истории, мало предметов) — только `flip` частично тестируем уже сегодня.

## 9. Формулировки про Steam-кошелёк

Из `docs/SCOPE.md`: деньги от продажи остаются в Steam-кошельке, не выводятся как реальные деньги. Формулировки в Telegram-боте (Этап 11) должны явно учитывать не только «это не финсовет», но и «это не вывод денег в банк». Далеко впереди по плану, но легко забыть — отмечено здесь именно для этого.

## 11. Дорожная карта реализации — порядок, в котором ничего не упускается

Порядок ниже — не переопределение этапов CLAUDE.md, а рекомендация, что́ разумно делать внутри Этапа 9 и вокруг него, с учётом реальных зависимостей (нельзя откалибровать веса без бэктеста, нельзя бэктестить без реализации, и т.д.):

1. **Сейчас, без кода, вне очереди:** тестовая покупка дешёвого предмета — подтвердить market hold. Ни на что не блокирует, но должно случиться до того, как flip-сигналам вообще начнут доверять деньгами.
2. **Этап 9, часть 1 — расчёт, без доверия к результату.** Реализовать `spread_score`, `trend_score`, `liquidity_factor`, `volatility_penalty`, `final_score` с явно помеченными плейсхолдер-весами (например, все `w = 1` или взятые «на глаз») — на данных по 3 уже отслеживаемым предметам. Включить подтверждение flip-отклонения на двух подряд опросах (раздел 7) сразу, а не откладывать — дёшево реализовать, дорого забыть. Definition of Done для этой части явно НЕ должен включать «сигналы можно слать» — только «score считается и пишется в таблицу».
3. **Этап 9, часть 2 (или отдельный «Этап 9.5») — бэктест.** Прогнать формулу из п.2 по истории `price_snapshots` для тех же 3 предметов: что было бы, если бы правило сработало в прошлом. Только после этого — калибровка весов `w1–w4` и порогов по фактическим результатам, не «на глаз».
4. **После, не до — расширение охвата.** Подключить `fetch_market_items` к автосбору (шире список предметов), заодно унифицировать разогрев сессии (раздел 5 — два механизма стоит свести к одному, раз файл всё равно открыт). Затем — повторный бэктест на расширенном наборе, потому что калибровка на 3 предметах не обязана обобщаться на весь рынок.
5. **Только теперь** — Signal Generator и Telegram Bot по исходному плану (Этапы 10–11), с формулировками из раздела 9.

Почему не наоборот (например, сначала расширить охват, потом считать score): расширение — это изменения в Collector, а не в Analyzer, и по правилам CLAUDE.md («изменять файлы, не относящиеся к текущему этапу») их лучше не смешивать с Этапом 9. И калибровать веса есть смысл один раз, по итогам бэктеста, а не дважды — один раз на 3 предметах, потом переделывать на большем наборе.
