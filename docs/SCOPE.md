# SCOPE.md — Договорённости и рамки проекта (Этап 0)

Зафиксировано в рамках Этапа 0. Меняется только по явному решению, не по ходу дела.

## Окружение разработки

- ОС разработки: Windows 11.
- Python: 3.11 (`C:\Program Files\Python311`). Изначально стоял 3.14.6, но при установке зависимостей (Этап 4) выяснилось, что под 3.14 нет готовых бинарных сборок для `pydantic-core` и `aiohttp` — потребовалась бы компиляция из исходников (Rust + MSVC Build Tools). Перешли на 3.11, где для всего стека есть готовые wheels.
- Виртуальное окружение проекта: `c:\Users\tokar\.virtualenvs\.venv` (пересоздано под Python 3.11).
- В системе настроен локальный SOCKS-прокси (реестр Windows, `127.0.0.1:10808`) — pip пытается через него подключаться и падает с ошибкой SOCKS-поддержки. Обход: перед установкой пакетов выставлять `$env:NO_PROXY = "*"` и `$env:no_proxy = "*"` в PowerShell-сессии.

## Источник данных Steam

- Steam Web API (официальный, по API-ключу, домен `api.steampowered.com`) — ключ добавлен в `.env` и проверен рабочим, но пока ни один этап плана его не использует (это отдельная система от Market, не даёт доступа к ценам).
- Для цен Market используется `priceoverview` (Этап 6) — полностью публичный эндпоинт `steamcommunity.com/market/`, не требует ни API-ключа, ни логина. Даёт только текущую цену и объём на момент запроса; свою историю копим сами со временем через регулярные вызовы (Этап 8).
- `pricehistory` (полная история цен от Steam за прошлое) первоначально был сознательно отложен как требующий cookie-сессии залогиненного аккаунта. Решение пересмотрено (см. запись ниже от 15.07.2026): используется для разового ручного backfill через `scripts/backfill_price_history.py`, с личным cookie разработчика, не как часть автоматического Collector'а.
- Соблюдаем разумные паузы между запросами (rate limit); без обхода антибот-защиты в смысле обмана систем безопасности (капчи, спуфинг отпечатков, ротация прокси ради нового лимита) и без агрессивного скрапинга молча, без обсуждения.
- На Этапе 8 при тестировании с интервалом 6 секунд поймали `429 Too Many Requests` от `priceoverview`. Блокировка держалась дольше 20 минут — реальный лимит Steam жёстче и "дольше отходит", чем предполагалось. Урок: тестировать со значительными паузами (минуты, не секунды) и не повторять запросы после первого `429`, пока не пройдёт заметно больше времени.
- Если для какой-то функции официального/документированного способа получения данных нет — вопрос обсуждается до реализации, а не решается агрессивным скрапингом молча.

**15.07.2026 — пересмотр после диагностики 429.** Выяснилось, что `priceoverview` и `search/render` режут "голые" запросы (без cookies и `Referer`) как подозрительные, даже с корректным `User-Agent`. Решение, принятое совместно и поэтапно (не молча): Collector сначала обычным GET-запросом заходит на страницу товара/поиска (как это делает любой анонимный посетитель сайта) и получает те же анонимные cookies, что получил бы браузер, затем добавляет `Referer` на эту страницу к основному запросу. Это не обход антибот-защиты и не спуфинг — запросы делают ровно то, для чего эндпоинт предназначен (обычный просмотр страницы товара пользователем), просто корректно, в отличие от прежнего "голого" вызова API в обход обычного пользовательского пути. Капча не обходится, отпечатки не подделываются, прокси не используются для смены "личности" ради лимита.

## Хостинг

- Пока прибыльность сервиса не подтверждена, всё разворачивается локально, на домашнем ПК — сервер/хостинг не покупаем.
- Telegram-бот работает через long-polling (не webhook), поэтому публичный IP/проброс портов не нужен.
- FastAPI не выставляется наружу в интернет, пока сервис используется только самим разработчиком.
- К вопросу о сервере возвращаемся, если: (а) подтвердится польза/прибыльность сервиса, и/или (б) понадобится работа сервиса, когда домашний ПК выключен, и/или (в) появятся другие пользователи.

## Позиционирование сервиса

- Это аналитический инструмент: сбор истории цен, статистический анализ, сигналы.
- Это не финансовая рекомендация и не лицензированная инвестиционная консультация.
- Во всех пользовательских интерфейсах (Telegram-бот, API, будущий UI) формулировки должны явно давать понять: результат — статистический сигнал, а не финансовый совет.
