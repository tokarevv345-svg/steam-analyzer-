## Этап 4 — Проектирование базы данных

Статус: готово (см. `src/steam_analyzer_data/storage/models.py`)

Сначала ER-диаграмма, затем обсуждение. Модели SQLAlchemy — только после подтверждения.

Итоговая схема: `items` (карточка предмета: market_hash_name, item_type, exterior, stattrak, rarity, image_url) 1—N `price_snapshots` (сырая история цен: price, volume, collected_at) и 1—N `signals` (сгенерированные сигналы: signal_type — flip/investment, цены, скоринг, sent_to_telegram). Валюта фиксирована как USD на уровне всей системы, отдельного поля под неё нет. Агрегаты (медиана/std) не хранятся отдельной таблицей — считаются SQL-запросами по `price_snapshots` на лету.

## Приложение А — критерии профита для будущих Analyzer/Signal Generator

Справочные заметки на будущее (Этапы 9-10). Логика здесь **не реализуется** на Этапе 4 — только зафиксирована текстом, чтобы не забыть.

**Для flip-сигнала:**

- `spread_score = (median_sell_price_7d − current_buy_price) / current_buy_price`
- `net_profit = sell_price * (1 − steam_fee_rate) − buy_price`, где `steam_fee_rate ≈ 0.13–0.15` (точное значение для CS уточнить отдельно)
- фильтр по ликвидности: минимум N продаж/день, иначе сигнал отсекается
- фильтр по волатильности: отсекать аномальные скачки (`std_dev/mean` выше порога)

**Для investment-сигнала:**

- сравнение скользящих средних MA7 vs MA30
- наклон линейной регрессии цены за N дней (`trend_score`)

**Общий скоринг** (черновая формула, веса подбираются позже эмпирически или через ML):

`final_score = w1*spread_score + w2*liquidity_factor − w3*volatility_penalty + w4*trend_score`

**Ограничения и риски, которые нужно учитывать при реализации:**

- это статистические эвристики по историческим данным, не гарантия будущего результата;
- перед тем как доверять сигналам, нужен бэктест на исторических данных;
- деньги от продажи остаются в Steam-кошельке и не выводятся как реальные деньги — формулировки в боте должны явно это учитывать (аналитический сигнал, а не финансовый совет).

## Незакоммиченная доработка — 17.07.2026 (fix 429 в Collector + Arbitrage-сигнал)

Не привязано к конкретному этапу из списка выше — правки по ходу разбора стороннего репозитория (steammarketscraper) на безопасность: выяснилось, что он бьётся в другой Steam-эндпоинт с намного более мягким лимитом запросов, и что это заодно даёт данные для честного bid/ask-арбитража. Написано в сессии с Claude (Cowork), ждёт проверки и явного подтверждения перед git-коммитом.

Что сделано:

- Collector переведён с `priceoverview` на `itemordershistogram` (`collector/steam_market_client.py`) — устраняет 429, который раньше прилетал меньше чем за 10 запросов.
- `Item.item_nameid` — кэш внутреннего числового ID предмета в Steam, резолвится один раз на предмет, а не на каждый цикл сбора.
- **Правка от 17.07.2026 после первой проверки в Claude Code:** первая версия `resolve_item_nameid()` (regex `Market_LoadOrderSpread` по HTML страницы листинга) не заработала — Steam с 2026 года отдаёт эту страницу через клиентский React/Next.js рендеринг, нужной разметки в ответе сервера больше нет вообще (подтверждено вживую: ни в HTML, ни в 86 статических JS-чанках страницы). Переписано на сторонний, вручную поддерживаемый датасет `github.com/somespecialone/steam-item-name-ids` (`data/CS2/item_names.json`, market_hash_name→item_nameid) — тем же самым пользуется автор референс-репозитория. Кэшируется в памяти процесса, не на диске. Риск: датасет неполный (собственное README репозитория предупреждает "Does not contain all items") — `resolve_item_nameid()` в этом случае кидает понятную `SteamMarketError`, а не падает необъяснимо.
- `PriceSnapshot.highest_buy_order` / `buy_order_count` — встречная сторона книги ордеров, раньше эти данные просто выбрасывались.
- `SignalType.ARBITRAGE` + `analyze_arbitrage()` в `analyzer/signal_calculator.py` — мгновенный bid/ask-сигнал по последнему снепшоту, без исторического окна. Порог `MIN_ARBITRAGE_NET_PROFIT_PCT = 5%` — заглушка, не откалиброван. Фильтр по глубине стакана (защита от тонких/поддельных стаканов) сознательно отложен — сначала накопить реальные `buy_order_count`, порог калибровать по факту.
- Мелкий mypy-баг из первой проверки (`item.item_nameid` типизирован как `int | None`, mypy не видит, что после `save_item_nameid()` он уже не `None`) — исправлен явной локальной переменной `nameid` в `collection_job.py`.
