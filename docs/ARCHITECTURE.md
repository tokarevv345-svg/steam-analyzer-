# ARCHITECTURE.md — Архитектура проекта (Этап 1)

Логическая архитектура на уровне модулей. Без структуры папок (Этап 2) и без схемы БД (Этап 4).

## Стиль архитектуры

Модульный монолит: один проект, один процесс деплоя, но с чётким разделением ответственности между внутренними модулями. Микросервисы не используются — избыточная сложность для текущего масштаба.

## Схема потоков данных

```
                    ┌────────────────────┐
                    │   Steam (внеш.)    │
                    └────────────────────┘
                               │ HTTP (Web API / price history)
                               ▼
          триггер
             │
             ▼
   ┌──────────────────┐     ┌──────────────────┐
   │    Scheduler     │     │    Collector     │
   │  (APScheduler)   │ ──▶ │  (сбор данных)   │
   └──────────────────┘     └──────────────────┘
                             │ сырые данные
                             ▼
                    ┌────────────────────┐
                    │      Storage       │
                    │    (Postgres +     │
                    │    SQLAlchemy)     │
                    └────────────────────┘
                               │ читает историю
                               ▼
                    ┌────────────────────┐
                    │      Analyzer      │
                    │    (статистика,    │
                    │    позже + ML)     │
                    └────────────────────┘
                               │ находки
                               ▼
                    ┌────────────────────┐
                    │  Signal Generator  │
                    │ (правила, формат)  │
                    └────────────────────┘
                               │ сигнал
                               ▼
                    ┌────────────────────┐
                    │    Telegram Bot    │
                    │     (aiogram)      │──▶ пользователь
                    └────────────────────┘

           читает/кэш
┌────────────────┐      ┌────────────────┐
│    FastAPI     │◀────▶│     Redis      │
│  (внеш. API)   │      │ (кэш + брокер) │
└────────────────┘      └────────────────┘
     │ читает                    │
     ▼                           ▼

┌────────────────┐   ┌────────────────┐
│    Storage     │◀──│ Celery worker  │
└────────────────┘   └────────────────┘
                     тяж. задачи (анализ, сигналы batch'ем)
```

## Модули и обоснование разделения

| Модуль | Отвечает за | Почему отдельно |
|---|---|---|
| **Collector** | только получение данных со Steam | Меняется по своим причинам (rate limit, формат ответа Steam) — не должен тянуть за собой анализ или БД-логику |
| **Storage** (Postgres/SQLAlchemy) | хранение истории цен | Единая точка правды о данных, не зависит от того, кто и как их анализирует |
| **Scheduler** (APScheduler) | запуск Collector по расписанию | Отдельно от Collector, чтобы можно было менять расписание, не трогая логику сбора |
| **Analyzer** | статистические расчёты по данным из Storage | Не знает про Telegram и про то, что будет с результатом дальше — только считает |
| **Signal Generator** | решает, достоин ли результат Analyzer стать сигналом, форматирует его | Отделён от Analyzer: завтра можно поменять правила отбора сигналов, не трогая расчёты |
| **Telegram Bot** (aiogram) | доставка сигнала пользователю | Просто канал доставки — завтра можно добавить email/webhook, не трогая Signal Generator |
| **FastAPI** | внешний API для чтения данных | Синхронный слой, не должен блокироваться на тяжёлых задачах |
| **Redis** | кэш для FastAPI + брокер для Celery | Переиспользуем одну инфраструктурную часть под две задачи — не плодим лишние зависимости |
| **Celery** | фоновые тяжёлые задачи | Снимает нагрузку с FastAPI/Scheduler, чтобы долгие операции не блокировали остальное |
| **ML-модуль** (Этап 17) | альтернативный источник оценок для Analyzer | Подключается как ещё одна стратегия внутри Analyzer, когда накопится история — не отдельный модуль верхнего уровня |
